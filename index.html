CASE 01
Nonclustered Index 신규 생성 — Key Lookup 제거
⚠ 왜 문제라 판단했나
UNION ALL로 동일 테이블 3회 참조 쿼리에서 Actual Plan 확인 시 매 실행마다 Key Lookup이 수십만 건 발생. Clustered Index만 존재해 조건절 컬럼 조회 후 데이터 페이지를 별도 탐색하는 구조가 I/O 폭증의 직접 원인이라 판단.
✓ 왜 이 방법을 선택했나
Key Lookup 제거에는 ① Clustered Key 변경, ② Nonclustered + INCLUDE 두 가지 선택지가 있었음. 운영 중 Clustered Key 변경은 리스크가 크므로 INCLUDE 절로 SELECT 컬럼을 커버링하는 Nonclustered Index를 설계. 선행 키는 조건절 중 선택도가 가장 높은 [REQDT]로 선정.
Logical Reads 99% 감소 498,553 → 39 page 305s → 1s
-- Clustered Index만 존재 → Key Lookup 반복 발생 SELECT order_id, product_cd, qty, amount FROM TBL_ORDER WHERE order_dt BETWEEN '20240101' AND '20240131' -- → Clustered Index Seek + Key Lookup × 수십만 건
-- Nonclustered Index: 선행키 order_dt + INCLUDE 컬럼 CREATE NONCLUSTERED INDEX IX_ORDER_DT ON TBL_ORDER (order_dt) INCLUDE (order_id, product_cd, qty, amount); -- → Index Seek only, Key Lookup 완전 제거 -- Logical Reads: 498,553 → 39 page (↓99%)
CASE 02
Cursor 제거 → Set 기반 처리 전환
⚠ 왜 문제라 판단했나
실행 계획에서 루프 반복 횟수가 처리 건수와 1:1로 증가하는 것을 확인. Cursor는 건당 UPDATE를 개별 실행하므로 데이터 증가 시 I/O·CPU가 선형으로 증가하는 구조적 문제. 부하 시간대와 Cursor 수행 시간이 정확히 일치하는 것으로 원인 특정.
✓ 왜 이 방법을 선택했나
Cursor 내 로직이 조건 분기 없이 단순 UPDATE임을 확인. JOIN 기반 Set 처리로 완전 대체 가능한 구조였기 때문에 재작성 선택. 변환 전후 결과 건수·값을 검증하여 정합성 확인 후 운영 적용.
반복 처리 완전 제거 I/O 대폭 감소 CPU 안정화
-- 행 단위 반복 처리 → 건수 비례 I/O·CPU 선형 증가 DECLARE @id INT, @status VARCHAR(10) DECLARE cur CURSOR FOR SELECT user_id, status FROM TBL_USER WHERE reg_dt = GETDATE() OPEN cur FETCH NEXT FROM cur INTO @id, @status WHILE @@FETCH_STATUS = 0 BEGIN UPDATE TBL_HISTORY SET status = @status WHERE user_id = @id FETCH NEXT FROM cur INTO @id, @status END CLOSE cur; DEALLOCATE cur
-- Set 기반: 단일 UPDATE로 일괄 처리 UPDATE H SET H.status = U.status FROM TBL_HISTORY H INNER JOIN TBL_USER U ON H.user_id = U.user_id WHERE U.reg_dt = GETDATE() -- → Cursor 제거, I/O·CPU 대폭 감소
CASE 03
불필요한 테이블 Join 제거
⚠ 왜 문제라 판단했나
실행 계획에서 TBL_CODE 테이블 Join 비용이 전체 I/O의 상당 부분을 차지하는 것을 확인. SELECT 절 전체를 검토한 결과 해당 테이블의 어떤 컬럼도 최종 결과에 사용되지 않음을 발견. 불필요한 Join이 매 실행마다 추가 I/O를 발생시키는 구조적 낭비로 판단.
✓ 왜 이 방법을 선택했나
Join 제거 시 결과에 영향이 없으려면 두 가지를 확인해야 함: ① 조인 키가 PK여서 중복 행 발생 없음, ② WHERE절에서도 해당 테이블 미사용. 두 조건 모두 만족함을 검증 후 안전하게 제거. 로직 변경 없이 성능 개선 가능한 가장 확실한 방법.
I/O 감소 실행 계획 단순화 로직 무변경
-- SELECT 절 미사용 테이블이 LEFT JOIN에 포함됨 SELECT S.sale_id, S.amount, S.sale_dt FROM TBL_SALE S LEFT JOIN TBL_CODE C -- SELECT·WHERE 미사용, PK Join ON S.code_id = C.code_id WHERE S.sale_dt >= '20240101'
-- Join 제거: PK Join + 결과 영향 없음 검증 후 적용 SELECT S.sale_id, S.amount, S.sale_dt FROM TBL_SALE S WHERE S.sale_dt >= '20240101' -- → 동일 결과 보장, I/O 감소, 실행 계획 단순화
CASE 04
인덱스 생성 + 중복 Join 제거 복합 튜닝
⚠ 왜 문제라 판단했나
실행 계획에서 동일 테이블(ZASCMT0060)이 조인마다 Full Scan 반복 발생. 해당 테이블에 인덱스가 없어 매번 전체 데이터를 읽는 구조. 추가로 같은 테이블을 2회씩 중복 조인하고 있어 불필요한 I/O가 배로 발생하고 있었음.
✓ 왜 이 방법을 선택했나
문제가 두 개(인덱스 부재 + 중복 Join)였기 때문에 각각 단계별로 효과를 측정한 뒤 함께 적용. 인덱스만 생성하면 중복 Join 비용이 남고, 중복 Join만 제거하면 Full Scan이 남음. 두 가지를 함께 해결해야 완전한 개선이 가능하다고 판단.

CASE 05
MAXDOP 조정 + 병렬 처리 쿼리 개선
⚠ 왜 문제라 판단했나
로그 백업과 ALTER INDEX REBUILD(maxdop=8)가 동일 시간대에 수행되며 CPU 100% 도달. Actual Plan에서 병렬 처리 스레드가 과다 할당된 것을 확인. 두 작업이 CPU 자원을 동시에 최대로 사용하는 구조적 충돌이 원인.
✓ 왜 이 방법을 선택했나
즉각적인 해결로 MAXDOP 조정 검토와 작업 시간대 분리 두 가지를 선택. MAXDOP 변경은 다른 쿼리에 영향을 줄 수 있어 신중히 검토하고, 근본 해결책으로 배치 작업을 서비스 peak 외 새벽 시간대로 이동하는 정책 수립.
CPU 100% → 안정화 MAXDOP 조정 배치 시간대 분리
CASE 06
백업 정책 개선 — 트랜잭션 로그 무한 증가 해결
⚠ 왜 문제라 판단했나
트랜잭션 로그 파일이 지속적으로 증가하는 것을 모니터링에서 감지. 백업 히스토리 조회 결과 Log Backup이 전혀 수행되지 않고 있었음. MSSQL Full Recovery Model에서 Log Backup 없이는 로그가 재사용되지 않아 무한 증가하는 구조적 문제.
✓ 왜 이 방법을 선택했나
Log Backup 도입이 유일한 근본 해결책. 단순 로그 축소(SHRINK)는 임시방편이고 재증가가 반복됨. Full Backup 빈도도 업무 시간대 다회 수행 중이어서 함께 개선. Full Backup 새벽 1회 + Log Backup 1시간 주기로 정책 수립 후 적용.
로그 파일 증가 차단 디스크 부하 완화 운영 안정화

⚠ 왜 이게 문제라 판단했나

MSSQL · ALTER TABLE DDL → Blocking 576건 조치 완료
⚠ 왜 이게 문제라 판단했나
Lock Waiting이 30건→576건으로 19배 폭증. MaxGauge Blocking 뷰에서 블로커 세션이 ALTER TABLE 수행 중임을 즉시 확인. DDL은 Object 레벨 잠금을 유지하므로 해당 테이블 접근 세션 전체가 대기하는 것.
✓ 왜 이 방법으로 해결했나
가장 빠른 해소는 DDL 세션 종료. 고객사 담당자와 즉시 협의 후 작업 중단. 재발 방지로 ALTER TABLE은 서비스 트래픽이 없는 시간대에 수행하도록 운영 정책 수립. 추가로 수행 전 해당 테이블 접근 세션 여부를 사전 확인하는 체크리스트를 제공하여 유사 상황 예방 체계 구성.

MSSQL · Lock Escalation + 대량 I/O Blocking 원인 분석 완료
⚠ 왜 이게 문제라 판단했나
Logical Reads 1,700만 건 SELECT 쿼리가 처리 지연되며 다른 세션 블로킹 유발. 인덱스 부재로 Full Scan 반복이 Row Lock을 과다 발생시키고, 이것이 Table 레벨로 에스컬레이션되는 구조적 원인 확인.
✓ 왜 이 방법으로 접근했나
Lock Escalation 자체보다 근본 원인인 Full Scan + 대량 Lock 발생 구조 개선이 필요. 인덱스 최적화로 처리 범위를 줄이고, 대용량 배치와 서비스 쿼리의 시간대를 분리하는 방향으로 권고.

Oracle · Redo Log I/O 병목 → 커밋 지연 연쇄 분석 수행
⚠ 왜 이게 문제라 판단했나
홈페이지 INSERT 세션 대기 급증. 단순 쿼리 문제가 아님을 의심하고 세션 대기 이벤트를 역방향으로 추적. log file sync → LGWR → log file parallel write 순으로 연쇄 구조를 확인. Redo I/O 자체가 병목임을 특정.
✓ 한계와 배운 점
병목 지점(log file parallel write)은 특정했으나 디스크 I/O 지표·alert log를 실시간으로 확보하지 못해 최종 원인 확정 불가. 이후 해당 지표들을 모니터링 수집 항목에 추가. OS 레벨 지표와 DB 대기 이벤트를 함께 봐야 한다는 것을 체감.

↓ INSERT 세션 블로킹 연쇄
